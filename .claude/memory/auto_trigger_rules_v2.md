# 自动触发规则 v2 - 深度细化版

## 核心设计理念

1. **上下文感知**: 根据对话历史、当前任务阶段、文件变更状态动态调整触发行为
2. **多阶段触发**: 任务开始前、执行中、完成后分别触发不同的辅助行为
3. **条件判断**: 根据项目状态（是否有未提交变更、是否首次创建等）决定执行路径
4. **最小干扰**: 只在必要时触发，避免过度自动化影响用户控制感
5. **可追溯**: 每次触发都记录原因和执行的行动，便于用户理解和调整

## 一、Go 后端开发触发规则

### 1.1 触发条件（满足任一即可）

**关键词匹配**:
- 高优先级（立即触发）: "创建 API"、"新增接口"、"写一个 Handler"、"添加路由"
- 中优先级（确认后触发）: "后端"、"Go"、"Gin"、"GORM"、"model"、"api"
- 低优先级（询问后触发）: "接口"、"数据"、"查询"、"保存"、"删除"

**上下文感知**:
- 用户刚创建了 PRD/需求文档 → 主动询问是否需要开发对应 API
- 用户刚修改了 database schema → 主动询问是否需要同步更新 model
- 用户刚完成了前端页面 → 主动询问是否需要开发对应 API

### 1.2 执行流程

```
阶段 1: 准备（触发后自动执行）
├─ 读取 docs/BACKEND.md（了解开发规范）
├─ 读取 docs/STANDARD.md（了解代码风格）
├─ 检查 app/admin/model/system/ 是否存在相关 model
│  └─ 不存在 → 先创建 model
├─ 检查 app/admin/api/system/ 是否存在相似 API 作为参考
└─ 检查 app/admin/router/system/ 是否已注册路由

阶段 2: 执行（用户确认后）
├─ 创建/修改 model 文件（遵循 GORM 规范）
├─ 创建/修改 api 文件（使用 R.ReturnSuccess 格式）
├─ 创建/修改 router 文件（使用 jwt.JWTAuthMiddleware）
└─ 更新 app/admin/router.go（注册新路由）

阶段 3: 验证（完成后自动执行）
├─ 运行 go build 检查编译错误
├─ 运行 go fmt 格式化代码
├─ 运行 go vet 检查潜在问题
└─ 提醒用户测试 API 功能
```

### 1.3 输出规范

- Model 文件：`app/admin/model/system/sys{模块名}.go`
- API 文件：`app/admin/api/system/{模块名}.go`
- Router 文件：`app/admin/router/system/{模块名}.go`
- 函数命名：`{Action}{ModuleName}`（如 ListUser, GetUser, CreateUser）
- 返回格式：统一使用 `R.ReturnSuccess(data)` 和 `R.ReturnFailMsg(msg)`

## 四、Git 操作触发规则

### 4.1 触发条件

**关键词匹配**:
- 高优先级（立即执行）: "提交"、"推送"、"git commit"、"git push"
- 中优先级（确认后执行）: "保存代码"、"上传"、"同步远程"
- 低优先级（询问后执行）: "搞定"、"完成了"、"做完了"

**状态感知**:
- 有未提交变更 → 提示用户提交
- 有未推送提交 → 提示用户推送
- 远程有更新 → 提示用户拉取

### 4.2 执行流程

```
阶段 1: 检查（触发后自动执行）
├─ 运行 git status 查看变更
├─ 运行 git diff 预览变更内容
├─ 运行 git log -3 查看最近提交
└─ 运行 git remote -v 确认远程仓库

阶段 2: 提交（用户确认后）
├─ 根据变更内容分析提交类型：
│  ├─ 新功能 → feat:
│  ├─ 修复 bug → fix:
│  ├─ 文档变更 → docs:
│  ├─ 代码重构 → refactor:
│  └─ 配置/工具 → chore:
├─ 生成 commit message（包含变更摘要）
├─ 运行 git add .
└─ 运行 git commit -m "type: message"

阶段 3: 推送（用户确认后）
├─ 运行 git pull origin main（避免冲突）
├─ 运行 git push origin main
└─ 提醒用户查看远程仓库
```

### 4.3 Commit Message 规范

```
feat: 移动卫生执法系统全栈开发
- 后端：新增 XX 模块 API
- 前端：新增 XX 管理页面
- 移动端：新增 XX 功能

fix: 修复 XX 模块查询异常

docs: 更新开发文档

refactor: 重构 XX 模块代码结构

chore: 更新依赖配置
```

## 五、代码审查触发规则

### 5.1 触发条件

**关键词匹配**:
- 高优先级（立即执行）: "审查代码"、"code review"、"检查代码"、"有问题吗"
- 中优先级（确认后执行）: "对不对"、"好不好"、"怎么样"、"规范吗"
- 低优先级（询问后执行）: "看看"、"检查"、" review"

**上下文感知**:
- 用户刚完成代码编写 → 主动询问是否需要审查
- 用户准备提交代码 → 主动提供审查服务
- 用户修改了关键文件（model/api/router）→ 主动检查规范

### 5.2 审查维度

```
Go 后端审查:
├─ 代码规范
│  ├─ 命名是否符合 Go 规范（驼峰命名）
│  ├─ 导出类型/函数首字母是否大写
│  └─ 错误处理是否完善
├─ 项目规范
│  ├─ 是否使用 R.ReturnSuccess 返回格式
│  ├─ 是否使用 jwt.JWTAuthMiddleware 认证
│  └─ 文件路径是否正确
├─ 安全检查
│  ├─ 是否有 SQL 注入风险（必须使用 GORM 参数化）
│  ├─ 是否有敏感信息泄露
│  └─ 是否有越权访问风险
└─ 性能检查
   ├─ 是否有 N+1 查询问题
   ├─ 是否有资源泄漏（defer close）
   └─ 是否有不必要的循环

Vue 前端审查:
├─ 代码规范
│  ├─ 命名是否符合 Vue 规范（kebab-case）
│  ├─ 组件是否注册正确
│  └─ 生命周期钩子使用是否正确
├─ 项目规范
│  ├─ 是否使用统一组件（pagination 等）
│  ├─ 是否添加 v-hasPermi 权限指令
│  └─ API 路径是否正确
├─ 安全检查
│  ├─ 是否有 XSS 风险（v-html 使用）
│  ├─ 是否有 CSRF 风险
│  └─ 用户输入是否转义
└─ 性能检查
   ├─ 是否有不必要的计算
   ├─ 是否使用 v-show 替代 v-if
   └─ 列表是否添加 key

UniApp 移动端审查:
├─ 代码规范
│  ├─ 是否使用 uview-ui 组件库
│  ├─ 页面是否在 pages.json 注册
│  └─ API 调用是否使用 request
├─ 安全检查
│  ├─ 是否有敏感信息存储
│  ├─ 网络请求是否 HTTPS
│  └─ 文件上传是否有大小限制
└─ 性能检查
   ├─ 图片是否压缩
   ├─ 列表是否分页/虚拟
   └─ 是否有内存泄漏风险
```

## 七、多阶段任务触发规则

### 7.1 全栈开发场景

当用户提出完整功能需求时（如"开发单位管理功能"），自动拆分为多阶段任务：

```
阶段 1: 需求分析
├─ 确认功能范围（CRUD？搜索？导入导出？）
├─ 确认字段定义
├─ 确认权限配置
└─ 生成任务清单

阶段 2: 数据库设计
├─ 设计表结构
├─ 生成迁移 SQL
└─ 执行变更

阶段 3: 后端开发
├─ 创建 model
├─ 创建 api
├─ 创建 router
└─ 注册路由

阶段 4: 前端开发（PC）
├─ 创建页面
├─ 创建 API 接口
└─ 配置菜单

阶段 5: 移动端开发（可选）
├─ 创建页面
├─ 创建 API 接口
└─ 配置路由

阶段 6: 测试验证
├─ 后端 API 测试
├─ 前端页面测试
└─ 集成测试

阶段 7: 提交发布
├─ 代码审查
├─ Git 提交
└─ 推送远程
```

### 7.2 任务追踪

- 每个阶段完成后询问用户是否继续下一阶段
- 支持用户指定从任意阶段开始
- 支持跳过某些阶段（如"只开发后端，前端自己写"）

## 九、触发优先级管理

### 9.1 优先级定义

| 级别 | 触发方式 | 示例 |
|------|---------|------|
| P0 - 立即触发 | 无需确认，直接执行 | "提交代码"、"审查这个文件" |
| P1 - 确认后触发 | 简要确认后执行 | "创建 API"、"开发页面" |
| P2 - 询问后触发 | 详细询问需求后执行 | "做个功能"、"处理一下" |
| P3 - 主动建议 | 仅提供建议，等待用户决定 | "是否需要..."、"建议..." |

### 9.2 冲突处理

- 多个触发条件同时满足 → 执行优先级最高的
- 用户明确说"不要自动..." → 跳过自动触发，仅响应式执行
- 用户连续否定两次自动行为 → 暂停该类自动触发，等待用户指令
