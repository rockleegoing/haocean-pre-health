### 1.2 核心设计理念

```

├── 生活饮用水

所有业务数据关联行业：
• 文书模板 → 关联行业分类
• 法律法规 → 关联行业分类
• 规范用语 → 关联行业分类
• 监管事项 → 关联行业分类
• 执法单位 → 关联行业分类
• 执法记录 → 关联行业 + 执法单位
```

### 1.4 技术方案选型

**1. Word 模板编辑（后台）**：
| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Spire.WordJS** | 纯前端、无需插件、功能完整 | 商业授权 | ⭐⭐⭐⭐⭐ |
| **PageOffice** | 支持模板套红、在线编辑 | 需后端配合 | ⭐⭐⭐⭐ |
| **docx.js + 后端** | 开源免费、灵活 | 编辑能力有限 | ⭐⭐⭐⭐ |

**推荐方案**：Spire.WordJS（纯前端体验最佳）+ docx4j（后端处理）

**2. 文档格式**：
- 模板存储：`.docx` 格式（OpenXML，易于程序处理）
- 填空项标记：`${变量名}` 或 `{{变量名}}`
- 生成文书：`.docx`（可编辑）+ `.pdf`（正式归档）

**3. 移动端打印**：
| 平台 | 方案 | 插件 |
|------|------|------|
| **App (Android)** | 原生插件 | 佳博蓝牙打印插件 |
| **App (iOS)** | BLE 蓝牙 | 原生插件 |
| **小程序** | 服务端转图片 | uni 蓝牙 API |

**4. PDF 转换**：
- 服务端：LibreOffice / Aspose.Words
- 客户端：docx-pdf 库（前端转换，质量一般）
- 推荐：服务端转换（质量好，稳定性高）

**5. 参考来源**：
- Word 文档模板在线编辑：[CSDN](https://blog.csdn.net/gitblog_00266/article/details/154630972), [Spire.WordJS](https://www.cnblogs.com/software-Development/p/19292761)
- uniApp 蓝牙打印：[DCloud 插件市场](https://ext.dcloud.net.cn/plugin?id=462), [佳博插件](https://juejin.cn/post/7146464559002288136)

## 二、设备与后端关系

### 2.1 设备激活流程

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 1. 首次启动  │ →  │ 2. 获取激活码 │ →  │ 3. 激活设备  │
│   App        │    │   后台管理   │    │   绑定设备   │
└──────────────┘    └──────────────┘    └──────────────┘
                           │                    │
                           │ 生成激活码         │ 验证激活码
                           │ 绑定执法人员       │ 返回 Token
                           ▼                    ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 6. 完成激活  │ ←  │ 5. 下载基础  │ ←  │ 4. 登录系统 │
│   进入首页   │    │    数据      │    │   验证通过 │
└──────────────┘    └──────────────┘    └──────────────┘
```

### 2.2 设备 - 用户绑定关系

| 层级 | 说明 | 存储位置 |
|------|------|---------|
| 设备 ID | 设备唯一标识（UUID） | App 本地 |
| 激活码 | 后台生成的 6-8 位激活码 | 后台数据库 |
| 执法人员 | 绑定到具体执法人员 | 后台数据库 |
| 所属单位 | 执法人员所属执法机构 | 后台数据库 |
| 管辖区域 | 可执法的地理区域 | 后台数据库 |

### 2.3 设备管理功能（后台）

| 功能 | 说明 |
|------|------|
| 设备列表 | 查看所有已激活设备（设备 ID、绑定人员、激活时间、最后同步时间、激活状态） |
| 激活码管理 | 生成激活码/批量生成/导出/设置有效期/禁用 |
| 激活码列表 | 查看已生成激活码（码值、状态、绑定设备、有效期、使用人） |
| 设备解绑 | 解除设备与人员的绑定（需管理员权限） |
| 设备状态 | 在线/离线/最后同步时间/激活状态/激活码有效期 |
| 数据清除 | 远程清除丢失设备数据（敏感操作，需审批） |
| 激活码禁用 | 禁用未使用或已泄露的激活码 |
| 激活记录 | 查看设备激活历史（激活时间、设备信息、绑定人员） |

### 2.4 激活码详细设计

**激活码格式**：
```
格式：6-8 位字母数字组合（大写字母 + 数字）
示例：ABCD1234、XY9K2M、2026ABC
生成规则：随机生成，保证唯一性
```

**激活码状态**：
| 状态 | 说明 | 可否使用 |
|------|------|---------|
| 未使用 | 刚生成，未绑定设备 | ✅ 可用 |
| 已使用 | 已绑定设备 | ❌ 不可用 |
| 已过期 | 超过有效期 | ❌ 不可用 |
| 已禁用 | 管理员手动禁用 | ❌ 不可用 |

**激活码有效期**：
- 默认有效期：30 天（可配置）
- 过期后自动失效，无法激活
- 支持设置永久有效

**激活码生成方式**：
```
单个生成：后台手动生成单个激活码
批量生成：一次性生成多个激活码（10/50/100 个）
导出功能：导出激活码列表为 Excel（码值 + 有效期 + 备注 + 状态）
```

**激活码与执法人员的绑定**：
```
1. 管理员生成激活码时，可指定或后续绑定执法人员
2. 一个激活码只能绑定一个执法人员
3. 一个执法人员可以有多个激活码（用于多设备）
4. 激活码过期后，绑定的设备需要重新激活
```

### 2.5 激活状态检查机制

**App 端检查**：
```
启动时检查：
1. 检查本地是否已激活
2. 检查激活码是否过期
3. 检查 Token 是否有效

未激活或过期处理：
1. 跳转到激活页面
2. 禁用所有执法功能
3. 仅保留激活码输入功能
```

**后端检查**：
```
API 请求拦截：
1. 检查设备激活状态
2. 检查激活码有效期
3. 未激活或过期返回 403

定期验证：
1. 每次同步数据时验证激活状态
2. 激活码过期前 7 天发送提醒
```

### 3.3 离线同步策略

| 数据类型 | 同步时机 | 同步方式 | 冲突处理 |
|---------|---------|---------|---------|
| 行业分类 | 启动时/WiFi 下 | 后台静默更新 | 以服务器为准 |
| 执法记录 | 网络恢复后 | 自动 + 手动 | 以本地为准 |
| 新增单位 | 网络恢复后 | 自动 + 手动 | 查重后合并 |
| 单位修改 | 网络恢复后 | 手动确认 | 提示冲突选择（见下方详细机制） |
| 用户信息 | 登录后/定期 | 后台静默更新 | 以服务器为准 |
| 角色权限 | 登录后/定期 | 后台静默更新 | 以服务器为准 |
| 部门信息 | 登录后/定期 | 后台静默更新 | 以服务器为准 |
| 法律法规 | 启动时/WiFi 下 | 后台静默更新 | 以服务器为准 |
| 规范用语 | 启动时/WiFi 下 | 后台静默更新 | 以服务器为准 |
| 监管事项 | 启动时/WiFi 下 | 后台静默更新 | 以服务器为准 |

## 五、功能模块详细设计（更新版）

### 5.1 登录与激活

**首次使用流程**：
```
┌─────────────────────────────────────────────────────────────────────┐
│  首次使用流程：                                                      │
│  1. 启动 App → 检查本地激活状态                                       │
│  2. 未激活 → 跳转激活页面 → 输入激活码                                │
│  3. 验证激活码（在线）→ 激活成功 → 设置登录密码                        │
│  4. 输入用户账号 + 密码 → 在线登录验证                                │
│  5. 登录成功 → 下载基础数据 + 同步登录信息到本地                        │
│  6. 进入首页                                                          │
└─────────────────────────────────────────────────────────────────────┘
```

**后续使用流程（离线登录）**：
```
┌─────────────────────────────────────────────────────────────────────┐
│  启动流程：                                                          │
│  1. 启动 App → 检查本地激活状态                                       │
│  2. 激活码已过期 → 跳转激活页面（禁用 APP 使用）                       │
│  3. 激活码正常 → 进入登录页面                                         │
│  4. 登录页面 → 读取本地同步的用户信息                                  │
│  5. 本地校验密码（ bcrypt/argon2 验证）                               │
│  6. 登录成功 → 进入首页                                               │
│  7. 后台尝试联网 → 检查数据更新（静默下载）                            │
└─────────────────────────────────────────────────────────────────────┘
```

**激活码验证时机**：
```
┌─────────────────────────────────────────────────────────────────────┐
│  激活码验证节点：                                                    │
│  1. App 启动时 → 检查本地激活状态 + 激活码有效期                       │
│  2. 登录前 → 验证激活码是否过期（本地检查）                           │
│  3. 联网时 → 同步验证激活码状态（后台接口）                           │
│  4. 激活码过期 → 跳转激活页面，禁用所有执法功能                        │
└─────────────────────────────────────────────────────────────────────┘
```

**离线登录详细流程**：
```
┌─────────────────────────────────────────────────────────────────────┐
│  登录前检查：                                                        │
│  1. 读取本地 device 表 → 检查激活状态                                 │
│  2. 检查 activate_code.expire_time → 是否过期                        │
│  3. 已过期 → 跳转激活页面，禁止登录                                   │
│  4. 正常 → 进入登录页面                                              │
├─────────────────────────────────────────────────────────────────────┤
│  本地登录校验：                                                      │
│  1. 用户输入账号 + 密码                                              │
│  2. 读取本地同步的 user 表 → 查找匹配账号                             │
│  3. 使用 bcrypt/argon2 验证密码                                       │
│  4. 验证成功 → 生成本地 Token → 进入执法单位选择页面                    │
│  5. 验证失败 → 提示密码错误 → 记录尝试次数                           │
├─────────────────────────────────────────────────────────────────────┤
│  登录信息同步（联网时）：                                            │
│  1. 检测网络可用 → 调用 /api/auth/sync-info                         │
│  2. 下载用户信息、角色权限、部门信息                                  │
│  3. 更新本地 user/role/dept 表                                       │
│  4. 后续登录使用最新数据                                              │
└─────────────────────────────────────────────────────────────────────┘
```

  `inspector_names` TEXT,  -- 执法人员姓名列表 (JSON)
  `dept_id` INTEGER,  -- 部门 ID
  `status` TEXT DEFAULT 'draft',  -- draft/pending/submitted/approved/archived
  `check_result` INTEGER,  -- 0:合格 1:不合格
  `problem_desc` TEXT,  -- 问题描述
  `handle_opinion` TEXT,  -- 处理意见
  `rectify_deadline` DATE,  -- 整改期限
  `remark` TEXT,  -- 备注说明
  `is_sync` INTEGER DEFAULT 0,  -- 是否已同步
  `sync_id` INTEGER,  -- 服务器 ID
  `create_by` INTEGER,  -- 创建人 ID
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `update_time` DATETIME
);

-- 证据材料表
CREATE TABLE `evidence` (
  `evidence_id` INTEGER PRIMARY KEY AUTOINCREMENT,
  `record_id` INTEGER,
  `type` TEXT,  -- photo/audio/video/document
  `file_path` TEXT,  -- 文件路径
  `file_name` TEXT,  -- 文件名称
  `file_size` INTEGER,  -- 文件大小（字节）
  `duration` INTEGER,  -- 时长（秒，录音/录像用）
  `thumbnail_path` TEXT,  -- 缩略图路径
  `description` TEXT,  -- 描述说明
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (`record_id`) REFERENCES `enforcement_record`(`record_id`)

**5.6.9 离线存储设计**

```sql
-- 法律法规表
CREATE TABLE `regulation` (
  `regulation_id` INTEGER PRIMARY KEY,
  `title` TEXT NOT NULL,  -- 法律名称
  `legal_type` TEXT,  -- 法律类型（法律/法规/规章/规范性文件/批复文件/标准）
  `supervision_types` TEXT,  -- 监管类型列表（JSON 数组）
  `publish_date` TEXT,  -- 发布日期
  `effective_date` TEXT,  -- 实施日期
  `issuing_authority` TEXT,  -- 颁发机构
  `content` TEXT,  -- 完整内容
  `version` TEXT,  -- 版本号
  `is_collected` INTEGER DEFAULT 0,  -- 是否收藏
  `download_time` DATETIME,  -- 下载时间
  `update_time` DATETIME,
  `sync_time` DATETIME  -- 最后同步时间
);

-- 章节目录表
CREATE TABLE `regulation_chapter` (
  `chapter_id` INTEGER PRIMARY KEY,
  `regulation_id` INTEGER,
  `chapter_no` INTEGER,  -- 章节号
  `chapter_title` TEXT,  -- 章节标题
  `content` TEXT,  -- 章节内容
  `sort_order` INTEGER,
  FOREIGN KEY (`regulation_id`) REFERENCES `regulation`(`regulation_id`)
);

-- 条款表
CREATE TABLE `regulation_article` (
  `article_id` INTEGER PRIMARY KEY,
  `chapter_id` INTEGER,
  `article_no` TEXT,  -- 条款号（如：第一条）
  `content` TEXT,  -- 条款内容
  `is_collected` INTEGER DEFAULT 0,  -- 是否收藏
  `sort_order` INTEGER,
  FOREIGN KEY (`chapter_id`) REFERENCES `regulation_chapter`(`chapter_id`)
);

-- 定性依据表
CREATE TABLE `legal_basis` (
  `basis_id` INTEGER PRIMARY KEY,
  `basis_no` TEXT,  -- 编号（如：001）
  `title` TEXT NOT NULL,  -- 标题
  `violation_type` TEXT,  -- 违法类型
  `issuing_authority` TEXT,  -- 颁发机构
  `effective_date` TEXT,  -- 实施时间
  `legal_level` TEXT,  -- 效级
  `clauses` TEXT,  -- 条款内容（JSON）
  `legal_liability` TEXT,  -- 法律责任
  `discretion_standard` TEXT,  -- 裁量标准
  `regulation_id` INTEGER,  -- 关联法律法规 ID
  `is_collected` INTEGER DEFAULT 0,  -- 是否收藏
  `update_time` DATETIME,
  `sync_time` DATETIME
);

-- 收藏表
CREATE TABLE `collection` (
  `collection_id` INTEGER PRIMARY KEY,
  `target_type` TEXT,  -- regulation/article/basis
  `target_id` INTEGER,
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 8.2 离线存储策略

| 数据类型 | 存储位置 | 清理策略 | 容量限制 |
|---------|---------|---------|---------|
| 单位数据 | SQLite | 不清理 | 无限制 |
| 执法记录 | SQLite + 文件系统 | 同步后保留 30 天 | 1GB |
| 照片证据 | 文件系统 | 同步后保留 30 天 | 500MB |
| 录音证据 | 文件系统 | 同步后保留 30 天 | 200MB |
| 录像证据 | 文件系统 | 同步后保留 7 天 | 500MB |
| 法律法规 | SQLite | 更新时替换 | 无限制 |
| 规范用语 | SQLite | 更新时替换 | 无限制 |
| 文书模板 | SQLite + 文件 | 更新时替换 | 50MB |

### 8.3 同步重试机制

```
┌─────────────────────────────────────────────────────────────────────┐
│  同步失败处理：                                                      │
│                                                                      │
│  1. 第一次失败 → 等待 5 秒后重试                                     │
│  2. 第二次失败 → 等待 30 秒后重试                                    │
│  3. 第三次失败 → 等待 5 分钟后重试                                   │
│  4. 第四次失败 → 标记为失败，等待用户手动同步                         │
│                                                                      │
│  失败记录处理：                                                      │
│  • 用户可查看同步失败记录                                            │
│  • 用户可手动重试单条/全部                                           │
│  • 失败记录保留 7 天                                                 │
└─────────────────────────────────────────────────────────────────────┘
```

### 13.2 头像离线存储方案

```sql
-- 用户表增加头像字段
CREATE TABLE `user` (
  `user_id` INTEGER PRIMARY KEY,
  `username` TEXT UNIQUE,
  `avatar` TEXT,  -- 头像文件路径（本地文件系统）
  `avatar_blob` BLOB,  -- 或存储 base64 编码的图片数据
  ...
);
```

**头像上传离线处理**：
```

```

### 13.3 双向同步详细机制

**同步触发时机**：
```
┌─────────────────────────────────────────────────────────────────────┐
│  同步触发机制：                                                      │
├─────────────────────────────────────────────────────────────────────┤
│  1. 登录成功后立即检测网络 → 有网则自动同步                          │
│  2. 网络状态变化：离线→在线时触发同步                                │
│  3. 用户手动刷新：下拉刷新/点击同步按钮                              │
│  4. 定时同步：每 30 分钟自动检测同步（WiFi 环境下）                    │
│  5. App 启动时：检测距离上次同步时间，超过 1 小时则同步               │
└─────────────────────────────────────────────────────────────────────┘
```

┌─────────────────────────────────────────────────────────────────────┐

│     ↓             ↓             ↓            ↓             ↓         │
│  regulation  standard_     supervision   document_     notice       │
│             language        item        template                     │
└─────────────────────────────────────────────────────────────────────┘

第 3 阶段：上报数据同步（App→后台，双向同步）
┌─────────────────────────────────────────────────────────────────────┐
│  ⑪ 新增单位 → ⑫ 修改单位 → ⑬ 执法记录 → ⑭ 个人头像 → ⑮ 个人信息      │
│      ↓            ↓            ↓            ↓             ↓          │
│   subject     subject     enforcement_   avatar        user         │

└─────────────────────────────────────────────────────────────────────┘
```

### 13.5 同步队列表设计

```sql
-- 同步队列表（详细版）
CREATE TABLE `sync_queue` (
  `queue_id` INTEGER PRIMARY KEY AUTOINCREMENT,
  `table_name` TEXT NOT NULL,  -- 表名（subject/user/document 等）
  `record_id` INTEGER NOT NULL,  -- 记录 ID
  `action` TEXT NOT NULL,  -- insert/update/delete
  `sync_type` TEXT DEFAULT 'app_to_server',  -- app_to_server 或 server_to_app
  `data` TEXT,  -- JSON 格式的数据
  `priority` INTEGER DEFAULT 0,  -- 优先级（0:普通 1:重要 2:紧急）
  `status` TEXT DEFAULT 'pending',  -- pending/success/failed/conflict
  `retry_count` INTEGER DEFAULT 0,  -- 重试次数
  `max_retry` INTEGER DEFAULT 3,  -- 最大重试次数
  `error_msg` TEXT,  -- 错误信息
  `server_id` INTEGER,  -- 服务器返回的 ID（insert 时用）
  `conflict_info` TEXT,  -- 冲突信息 JSON（冲突时用）
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `update_time` DATETIME,
  `sync_time` DATETIME,  -- 最后同步时间
  INDEX `idx_status` (`status`),
  INDEX `idx_priority` (`priority`),
  INDEX `idx_sync_type` (`sync_type`)
);
```

**优先级规则**：
| 数据类型 | 优先级 | 说明 |
|---------|-------|------|
| 执法记录（已提交） | 2-紧急 | 优先上报 |
| 新增单位 | 1-重要 | 次优先 |
| 个人信息修改 | 1-重要 | 次优先 |
| 头像上传 | 0-普通 | 可延后 |
| 单位修改 | 0-普通 | 可延后 |

### 13.6 冲突处理详细机制

```
┌─────────────────────────────────────────────────────────────────────┐
│  冲突检测流程：                                                      │
├─────────────────────────────────────────────────────────────────────┤
│  1. 比较本地 version 与服务器 version                                  │
│     • version 相同 → 无冲突                                         │
│     • version 不同 → 进入下一步                                     │
├─────────────────────────────────────────────────────────────────────┤
│  2. 比较 update_time                                                │
│     • 本地 update_time > 服务器 update_time → 本地更新，可能冲突     │
│     • 本地 update_time < 服务器 update_time → 服务器更新，可能冲突   │
├─────────────────────────────────────────────────────────────────────┤
│  3. 逐字段比较值                                                    │
│     • 所有字段值相同 → 无冲突（只是时间戳不同）                     │
│     • 存在字段值不同 → 标记冲突字段                                 │
├─────────────────────────────────────────────────────────────────────┤
│  4. 生成冲突报告                                                    │
│     {                                                              │
│       "has_conflict": true,                                         │
│       "conflict_fields": ["address", "contact_phone"],              │
│       "local_value": {"address": "XX 路 100 号"},                     │
│       "server_value": {"address": "XX 路 100 号 A"},                 │
│       "resolution_suggestion": "user_choose"                        │
│     }                                                               │
└─────────────────────────────────────────────────────────────────────┘
```

**冲突解决策略**：
| 数据类型 | 默认策略 | 用户可选项 |
|---------|---------|-----------|
| 单位信息 | 以后台为准 | 保留本地/保留后台/手动合并 |
| 个人信息 | 以本地为准 | 保留本地/保留后台 |
| 执法记录 | 以本地为准 | 保留本地（强制） |
| 头像 | 以本地为准 | 保留本地（强制） |

### 13.9 离线登录增强

**离线登录流程**：
```
┌─────────────────────────────────────────────────────────────────────┐
│  离线登录详细流程：                                                  │
├─────────────────────────────────────────────────────────────────────┤
│  1. 启动 App → 检查本地激活状态                                       │
│     • 读取 device 表 → 检查 activate_code.expire_time                │
│     • 已过期 → 跳转激活页面，禁止登录                                │
│     • 正常 → 进入登录页面                                           │
├─────────────────────────────────────────────────────────────────────┤
│  2. 用户输入账号 + 密码                                              │
│     • 读取本地 user 表 → 查找匹配 username                            │
│     • 未找到 → 提示"账号不存在，请检查网络后重试"                     │
│     • 找到 → 使用 bcrypt 验证 password_hash                          │
├─────────────────────────────────────────────────────────────────────┤
│  3. 密码验证                                                        │
│     • 验证失败 → 提示密码错误 → 记录尝试次数（超过 5 次锁定 30 分钟）    │
│     • 验证成功 → 生成本地 Token（JWT）→ 进入执法单位选择页面          │
├─────────────────────────────────────────────────────────────────────┤
│  4. 网络检测（登录后）                                              │
│     • 调用 /api/health 检测网络                                      │
│     • 有网络 → 触发双向同步流程 → 显示同步进度条                      │
│     • 无网络 → 直接进入单位选择页（使用本地缓存数据）                 │
└─────────────────────────────────────────────────────────────────────┘
```

**3. 项明细结构更新**：

#### 18.3.2 监管事项模块更新

### 18.4 数据库设计更新

根据设计图分析，规范用语的数据库设计需要支持四级层级结构：

```sql
-- 规范用语主表
CREATE TABLE `standard_language` (
  `id` INTEGER PRIMARY KEY,
  `supervision_type_id` INTEGER,  -- 监管类型 ID
  `supervision_type_name` TEXT,   -- 监管类型名称
  `level1_category` TEXT,         -- 一级分类（如：放射卫生技术服务）
  `level2_category` TEXT,         -- 二级分类（如：资质认可）
  `item_name` TEXT,               -- 具体项名称（如：超资质认可或者批准范围）
  `item_code` TEXT,               -- 项编码
  `sort_order` INTEGER,           -- 排序
  `is_enabled` INTEGER DEFAULT 1, -- 是否启用
  `sync_time` DATETIME            -- 最后同步时间
);

-- 规范用语内容表
CREATE TABLE `standard_language_content` (
  `content_id` INTEGER PRIMARY KEY,
  `item_id` INTEGER,              -- 关联项 ID
  `clause_text` TEXT,             -- 条款内容
  `basis_clause` TEXT,            -- 依据条款
  `legal_liability` TEXT,         -- 法律责任
  `discretion_standard` TEXT,     -- 裁量标准
  `related_regulations` TEXT,     -- 关联法律法规（JSON 数组）
  `update_time` DATETIME,
  `sync_time` DATETIME
);
```

### 18.5 API 接口更新

根据设计图，规范用语模块的 API 接口需要支持层级查询：

```
# 获取监管类型列表
GET /api/standard-language/supervision-types

# 获取一级分类列表（按监管类型）
GET /api/standard-language/level1-categories?supervision_type_id=xxx

# 获取二级分类列表（按一级分类）
GET /api/standard-language/level2-categories?level1=xxx

# 获取具体项列表（按二级分类）
GET /api/standard-language/items?level2=xxx

# 获取项明细内容
GET /api/standard-language/item-detail?id=xxx

# 搜索规范用语
GET /api/standard-language/search?keyword=xxx
```

## 十九、分阶段实施计划

### 19.1 项目概述

**开发目标**：基于 Haocean HealthEnforcement 项目开发移动卫生执法系统，实现卫生执法人员的移动办公能力，支持离线操作、数据同步、快速出证等核心功能。

**技术栈**：
| 层级 | 技术选型 |
|------|---------|
| **后端** | Go 1.24+ + Gin + GORM v1.31.1 + MySQL 8.0/SQLite |
| **移动端** | uni-app 2.0.31+ + uview-ui + SQLite 离线存储 |
| **认证** | JWT (golang-jwt/jwt) + bcrypt 密码加密 |
| **文档生成** | docx4j / LibreOffice (后端 PDF 转换) |
| **蓝牙打印** | 佳博插件 / iOS BLE |

- 完整的后台管理系统（用户、角色、菜单、部门、字典、通知公告等）

- 移动端现有功能：登录、首页、工作台、通知公告、个人中心
- JWT 认证中间件、统一响应格式、数据库封装

**开发范围**：
- 后端：行业分类、单位管理、文书模板、法律法规、规范用语、监管事项、设备激活、数据同步
- 移动端：设备激活、数据同步、快速出证、执法记录、离线存储
